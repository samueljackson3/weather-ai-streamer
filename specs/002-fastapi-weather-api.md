Build a real, working Weather API using FastAPI that fetches live data from OpenWeather. ## Technical Design

### File Structure (output of this session)

```
weather-ai-streamer/
├── src/
│   ├── config.py       # Settings loaded from .env via pydantic-settings
│   ├── models.py       # Pydantic request/response schemas
│   └── main.py         # FastAPI app: routes, dependency injection, startup
├── .env                # OPENWEATHER_API_KEY (already exists)
└── specs/
    └── 002-fastapi-weather-api.md   # This file
```

> Note: `src/` is a flat structure — no sub-packages yet. Session 3 will add `ollama_client.py` here.

### Request Lifecycle

```
Browser / curl
      │
      ▼
FastAPI Router (main.py)
      │  → Validates path/query params via type hints (auto)
      │  → Injects httpx.AsyncClient via Depends()
      │
      ▼
fetch_weather(city, units, client)  ← async function
      │  → Calls OpenWeather REST API via httpx
      │  → Parses response into WeatherResponse (Pydantic validates)
      │  → Raises HTTPException on 404, 401, 5xx
      │
      ▼
WeatherResponse (models.py)
      │  → Serialized to JSON by FastAPI
      │
      ▼
HTTP 200 JSON response
```

### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| HTTP client | `httpx.AsyncClient` | Async-native; faster than `requests` in async context |
| Settings | `pydantic-settings` | Auto-loads `.env`, validates types, same Pydantic DX |
| HTTP client lifecycle | FastAPI lifespan + `Depends()` | One client shared across requests; proper teardown |
| Response model | Pydantic `BaseModel` | Auto-validation + auto-docs + IDE hints |
| Error strategy | `HTTPException` | FastAPI-native; translates to correct HTTP status codes |
| No router separation | Flat `main.py` | Unnecessary complexity at this stage; add in Session 6 polishing |

## API Contract

### `GET /health`

Simple liveness check. Always responds 200 if the server is running.

**Response** (200):
```json
{
  "status": "ok",
  "version": "0.1.0"
}
```

---

### `GET /weather/{city}`

Fetches current weather from OpenWeatherMap and returns a cleaned, typed response.

**Path Parameters**:
- `city` (string, required): City name (e.g., `"Seattle"`, `"London"`)

**Query Parameters**:
- `units` (string, optional, default: `"metric"`): `"metric"` | `"imperial"` | `"standard"`

**Response** (200):
```json
{
  "city": "Seattle",
  "country": "US",
  "temperature": 15.5,
  "feels_like": 14.2,
  "description": "overcast clouds",
  "humidity": 80,
  "wind_speed": 3.1,
  "units": "metric"
}
```

**Error Responses**:

| Status | Condition | Detail message |
|--------|-----------|----------------|
| 404 | City not found (OWM returns `cod: "404"`) | `"City 'Atlantis' not found"` |
| 401 | Invalid API key | `"Invalid OpenWeather API key"` |
| 422 | Invalid `units` value | Auto-generated by FastAPI via enum validation |
| 503 | OpenWeather unreachable / timeout | `"Weather service unavailable"` |

## Code Contract

### `src/config.py`

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    openweather_api_key: str
    openweather_base_url: str = "https://api.openweathermap.org/data/2.5"
    ollama_url: str = "http://localhost:11434"  # Ready for Session 3
    app_version: str = "0.1.0"

    model_config = {"env_file": ".env"}

# Module-level singleton — import this in other modules
settings = Settings()
```

**Key learning**: `pydantic-settings` reads `.env` automatically. If `OPENWEATHER_API_KEY` is missing, it raises a clear error at startup — not at the first request. This is "fail fast."

---

### `src/models.py`

```python
from pydantic import BaseModel
from enum import Enum

class Units(str, Enum):
    metric = "metric"
    imperial = "imperial"
    standard = "standard"

class WeatherResponse(BaseModel):
    city: str
    country: str
    temperature: float
    feels_like: float
    description: str
    humidity: int
    wind_speed: float
    units: Units

class HealthResponse(BaseModel):
    status: str
    version: str

class ErrorResponse(BaseModel):
    detail: str
```

**Key learning**: `Units` as a `str` Enum means FastAPI validates the query param automatically and lists valid options in `/docs`.

### `src/main.py` — Key Signatures

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI, HTTPException, Depends, Query
import httpx

# Lifespan: creates one shared httpx client for the app's entire lifetime
@asynccontextmanager
async def lifespan(app: FastAPI):
    async with httpx.AsyncClient() as client:
        app.state.http_client = client
        yield
    # Client closes here when app shuts down

app = FastAPI(title="Weather AI Streamer", version=settings.app_version, lifespan=lifespan)

# Dependency: extracts the shared client from app.state
async def get_http_client(request: Request) -> httpx.AsyncClient:
    return request.app.state.http_client

# Route: path param + optional query param with default
@app.get("/weather/{city}", response_model=WeatherResponse)
async def get_weather(
    city: str,
    units: Units = Query(default=Units.metric),
    client: httpx.AsyncClient = Depends(get_http_client),
) -> WeatherResponse:
    ...

# Core business logic (testable in isolation)
async def fetch_weather(
    city: str,
    units: Units,
    client: httpx.AsyncClient,
) -> WeatherResponse:
    """
    Calls OpenWeather API, parses response into WeatherResponse.
    Raises HTTPException (404, 401, 503) on failure.
    """
    ...
```

**Key learning**: `fetch_weather` is a plain async function — not a route handler. This keeps business logic testable without spinning up a full FastAPI app.

## Learning Progression (Micro-Steps)

Work through these in order. Don't skip ahead.

**Step A** (~20 min) — Minimal FastAPI app  
Start with the smallest possible working server: one `GET /health` route. Run it, curl it, open `/docs`. Feel the magic before adding complexity.

**Step B** (~20 min) — Pydantic models  
Create `models.py` with `WeatherResponse`. Write it, import it in a scratch file, instantiate it with dummy data. Try passing a string where a float is expected — observe the error Pydantic gives you.

**Step C** (~20 min) — Config  
Create `config.py`. Call `settings = Settings()` with `OPENWEATHER_API_KEY` missing from `.env` — what happens? Then restore it.

**Step D** (~30 min) — First real endpoint (no httpx yet)  
Add `GET /weather/{city}` but return hardcoded `WeatherResponse` data. Focus on the routing, path params, and response model — before adding the external API call.

**Step E** (~40 min) — Add httpx  
Add the `lifespan` context manager, `get_http_client` dependency, and real OpenWeather call. Test with a real city. Then pass a bad city name and a bad API key.

**Step F** (~20 min) — Error handling  
Map OpenWeather error codes to `HTTPException`. Verify each error case from the API Contract table.

**Step G** (~20 min) — Explore `/docs`  
Spend time with the auto-generated docs. Try submitting requests via the UI. Check that all responses, params, and error codes are documented.

## Rails Developer Notes

| Concept | Rails equivalent | Key difference |
|---------|-----------------|----------------|
| `pydantic-settings` BaseSettings | `Rails.application.config` + `dotenv-rails` | Pydantic validates types at startup; Rails just reads strings |
| Pydantic `BaseModel` (input) | `params.require().permit()` | Pydantic is a typed object, not a hash |
| Pydantic `BaseModel` (output) | `ActiveModel::Serializers` or `as_json` | Defined once as a class, auto-generates JSON schema |
| `HTTPException(status_code=404)` | `render json: ..., status: :not_found` | FastAPI raises an exception; Rails renders in controller |
| `Depends(get_http_client)` | `before_action :set_client` | DI is more explicit and composable; easier to test |
| `@app.get("/weather/{city}")` | `get "/weather/:city", to: "weather#show"` | Route + handler merged in one decorator |
| `Units(str, Enum)` | No direct equivalent | Rails would use `validates :units, inclusion:` — Pydantic does this at the type level |
| Auto `/docs` page | `rails routes` | FastAPI generates an interactive Swagger UI with zero config |

## Implementation Notes (April 9, 2026)

### What Worked Well ✅

1. **Spec-Driven Approach**: Knowing the end goal (real API calls, error handling) helped prioritize steps and avoid rabbit holes.

2. **Progressive Micro-Steps (A→F)**: Each step built on the previous. Separating "hardcoded response" (Step D) from "real API call" (Step E) made it clear where the integration happens.

3. **Production-Ready from Day 1**: Adding error handling and input validation at the start (not as an afterthought) taught what real code looks like, not toy code.

4. **Concept Checks with Learning**: Asking "Why does this work?" forced deep understanding, not just copy-paste coding. When answers were incomplete, explaining why the full answer matters was crucial.

5. **Type Hints as Documentation**: Seeing how `-> WeatherResponse` automatically powered `/docs` and IDE hints was a magic moment.

6. **Configuration as Code**: Using `pydantic-settings` for `.env` loading demonstrated the "fail fast" principle concretely: app refuses to start if config is missing, not crashes at runtime.

### Improvements for Future Runs ⚠️

1. **Better Step E Breakdown**: Real API integration is dense. Future runs should split it:
   - **E1** (~20 min): Simple `fetch_weather()` call, no error handling
   - **E2** (~20 min): Add error handling for 404, 401, 500s
   - **E3** (~10 min): Test with real cities and error cases
   
   This lets students see the happy path first, then build robustness.

2. **Teach `**kwargs` Unpacking Earlier**: Before introducing Pydantic, spend 2-3 minutes on `**data`:
   ```python
   # Show this equivalence:
   data = {"status": "ok", "version": "0.1.0"}
   HealthResponse(**data)           # Unpacking
   HealthResponse(status="ok", version="0.1.0")  # Same thing
   ```
   This demystifies Pydantic's magic.

3. **Document macOS SSL Certificate Issue**: Add to setup section:
   > "On macOS, you may get SSL certificate errors. Add `VERIFY_SSL=false` to `.env` for development. (Production uses `VERIFY_SSL=true`.)"

4. **Earlier Environment Awareness**: Move SSL verification config to Step C, not Step F. Show early that configuration is environment-specific:
   ```
   .env (dev):  VERIFY_SSL=false
   .env (prod): VERIFY_SSL=true   # (or use CI/CD secrets)
   ```

5. **Better Error Testing Checklist**: Add explicit "test these scenarios" section:
   - [x] Empty city name → 400
   - [x] City with whitespace → should be stripped & work
   - [x] City too long (50+ chars) → 400
   - [x] Non-existent city → 404 with helpful message
   - [x] Invalid units value → 422 (FastAPI enum rejection)
   - [x] Request timeout → 503 with timeout message
   - [x] Network error (kill Ollama internet) → 503 with network error message

6. **Code Comments Explaining FastAPI Magic**: Add inline comments explaining:
   ```python
   @app.get("/weather/{city}", response_model=WeatherResponse)
   # ↑ response_model=... means:
   #   1. Validates return value matches schema
   #   2. Auto-strips extra fields  
   #   3. Powers `/docs` endpoint
   #   4. Helps IDE type checking
   ```

7. **Concept Check: When Answers Are Incomplete**: The communication style of asking questions was great. But when the answer was incomplete (e.g., "wrong type" instead of "SSL verification is security risk"), we should always explain:
   - **Why** the full answer matters
   - **What goes wrong** with the incomplete understanding
   - **Example** of the failure mode
   
   Example response template:
   ```
   ✅ Right direction! Here's the complete picture:
   
   Your answer: "SSL verification disabled prevents security risks"
   Full story: SSL verification is a SECURITY RISK when disabled because:
   - MITM attackers can intercept HTTPS traffic
   - You won't catch forged certificates
   - In production, this would expose your API key
   
   That's why we use: VERIFY_SSL=true in prod, false only in local dev
   ```

8. **Lifespan/Dependency Injection Preview**: Briefly mention in Step E:
   > "Note: We're creating a new client in each request (not optimal). Session 4 will use a `lifespan` context manager to create one client shared across all requests."
   
   This motivates the architecture change and connects sessions.

### Testing Approach Lessons

- Manual curl testing is good for understanding HTTP + API contracts
- Adding pytest-asyncio mocking for `fetch_weather()` should come in Session 5 (testing session)
- Testing error handling is crucial earlier than expected (we did it in Step F/manual tests, should be earlier)

### Code Quality Observations

- Separation of `fetch_weather()` business logic from `get_weather()` route handler was **essential** — Session 3 will confirm by importing and reusing it
- Input validation (city length checks) should probably move to `models.py` using Pydantic `Field()` with constraints, not in the route handler
  ```python
  class CityQuery(BaseModel):
      city: str = Field(..., min_length=2, max_length=50)
  ```
  This keeps validation rules in the schema, not scattered in routes.

### What Students Should Understand by End

1. **Type hints are contracts**: They drive validation, docs, and IDE hints
2. **Fail fast principle**: Config errors at startup, not at first request
3. **Separation of concerns**: Business logic (testable) vs HTTP handling (framework-specific)
4. **Error handling hierarchy**: Validate early, provide clear messages, make debugging easy
5. **Async I/O**: `httpx.AsyncClient` doesn't block waiting for responses (unlike `requests`)
6. **Auto-generated docs**: Zero manual work with type hints + Pydantic

### Stretch Goals for Advanced Students
